## 本章导读

欢迎来到产品经理系统学习教程的第五章。如果说前几章帮你建立了"产品思维"，
那么这一章，将教你把思维**落地成文档** — 也就是产品经理吃饭的家伙：**PRD**。

---

## 什么是 PRD？

**PRD（Product Requirements Document）**，中文叫「产品需求文档」，是产品经理
用来向研发团队、设计团队、测试团队等协作方**准确传达产品需求**的核心文档。

简单来说：

> PRD 就是你脑袋里那个产品想法的"施工图纸"。
> 没有图纸，工人不知道怎么盖楼；没有 PRD，工程师不知道怎么写代码。

---

## 为什么 PRD 是 PM 最重要的输出物之一？

很多新人会问："PM 的核心价值到底是什么？"

答案很多，但如果只能选一个可量化的产出，大多数团队会指向 **PRD**。

```
+-------------------------------------------------------------+
|                   PM 的日常产出物                              |
+-------------------------------------------------------------+
|                                                             |
|   +----------+   +---------+   +---------+   +----------+  |
|   | 竞品分析  |   |  用户   |   |  数据   |   |  会议    |  |
|   |  报告    |   |  调研   |   |  分析   |   |  纪要    |  |
|   +----+-----+   +----+----+   +----+----+   +----+-----+  |
|        |              |             |              |         |
|        +--------------+------+------+--------------+         |
|                              |                              |
|                        +-----v-----+                        |
|                        |           |                        |
|                        |    PRD    |  <--- 所有工作的汇聚点   |
|                        |           |                        |
|                        +-----+-----+                        |
|                              |                              |
|              +---------------+----------------+             |
|              |               |                |             |
|        +-----v-----+  +-----v-----+  +-------v-------+    |
|        |  研发开发   |  |  UI 设计  |  |   QA 测试     |    |
|        +-----------+  +-----------+  +---------------+    |
|                                                             |
+-------------------------------------------------------------+
```

### PRD 的核心价值

| 价值维度 | 说明 |
|---------|------|
| **对齐认知** | 让团队所有人对"要做什么"达成一致理解 |
| **减少返工** | 需求描述越清楚，开发过程中的来回沟通越少 |
| **留下记录** | 产品决策有据可查，避免"他说她说"的扯皮 |
| **驱动协作** | 研发看功能需求、设计看交互说明、QA 看验收标准 |
| **衡量成果** | PRD 中的成功指标是后续复盘的基准线 |

## PRD 在产品开发流程中的位置

```
  需求发现          需求分析          需求文档化         研发实施          上线验收
                                                                    
 +--------+      +----------+      +----------+      +--------+      +--------+
 | 用户反馈 | ---> | 可行性   | ---> |  撰写    | ---> | 开发 & | ---> | 验收 & |
 | 市场调研 |      | 分析评估  |      |  PRD    |      | 设计   |      | 上线   |
 | 老板需求 |      | 优先级排序 |      |         |      |        |      |        |
 +--------+      +----------+      +----------+      +--------+      +--------+
                                        ^
                                        |
                                   你现在学的！
```

PRD 处于**需求分析**和**研发实施**之间，是承上启下的关键环节。

- **往上看**：PRD 是对业务需求（BRD）、市场需求（MRD）的细化落地
- **往下看**：PRD 是研发排期、UI 设计、QA 编写测试用例的直接依据

## 学习建议

1. **先通读，再动手**：先把三篇文章都读一遍，建立整体认知
2. **用模板练习**：拿第 02 篇的模板，给一个你熟悉的产品功能写一份 PRD
3. **找人 Review**：写完后找朋友（最好是工程师）帮你看，能不能看懂
4. **反复迭代**：好的 PRD 不是一次写成的，是改出来的

> **给零基础转行者的话**：
>
> 不要被 PRD 的"标准结构"吓到。刚开始的时候，你不需要写出完美的文档。
> 重要的是：**把需求说清楚**。格式和规范是锦上添花，清晰表达才是核心。
> 随着你写得越多、踩的坑越多，你的 PRD 自然会越来越好。

## PRD 的前世今生：一段简短的历史

在互联网行业兴起之前，"需求文档"更多存在于传统软件工程领域，
那时候叫做 SRS（Software Requirements Specification），按照 IEEE 标准来写，
动辄上百页，格式死板、审核流程冗长。

随着互联网和敏捷开发（Agile）的兴起，需求文档逐渐演变：

```
  传统 SRS 时代                 互联网 PRD 时代
  (2000年前)                   (2010年后)
                              
  +------------------+         +------------------+
  | 上百页的正式文档   |         | 精简的在线文档     |
  | 瀑布式开发       |   --->  | 敏捷迭代开发       |
  | 写完才能开发      |         | 边写边开发         |
  | 一次性交付        |         | 持续交付           |
  | 严格的评审流程     |         | 轻量的评审         |
  +------------------+         +------------------+
```

今天我们说的 PRD，是适应互联网快节奏开发的产物。它比传统 SRS 更灵活，
但依然保持了需求文档"说清楚做什么"的核心使命。

## 真实工作中，PRD 到底有多重要？

来看看真实场景中，有没有 PRD 的区别：

### 场景一：没有 PRD

```
  PM (口头):    "我们做个购物车功能吧"
  工程师:       "好的，大概什么样？"
  PM:          "就参考淘宝的"
  工程师:       "行，我做"
  
  ...... 两周后 ......
  
  PM:          "这不是我想要的！你怎么没做批量删除？"
  工程师:       "你没说要做啊！"
  PM:          "我觉得这是常识......"
  工程师:       "......"
  
  结果：返工 5 天，项目延期一周，团队关系紧张
```

### 场景二：有 PRD

```
  PM:          "这是购物车的 PRD，请大家评审"
  工程师:       "PRD 第 5.3 节，批量删除的逻辑我有个建议......"
  PM:          "好的，我更新一下 PRD"
  
  ...... 两周后 ......
  
  工程师:       "按 PRD 开发完了，你验收一下"
  PM:          "验收通过，和 PRD 描述一致"
  
  结果：按时交付，团队高效协作
```

这就是 PRD 的价值 — 它不是"走形式"，而是**真正减少沟通成本、避免返工**的工具。
