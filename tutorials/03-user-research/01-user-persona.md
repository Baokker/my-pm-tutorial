# 01 - 用户画像（Persona）

> "如果你试图取悦所有人，你最终谁也取悦不了。"

---

## 一、什么是用户画像

用户画像（Persona）是基于真实用户数据创建的**虚构但典型的用户原型**。
它不是某一个具体的人，而是一类用户群体的"代表人物"。

简单来说，用户画像就是回答一个核心问题：**我们的产品到底是做给谁用的？**

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   真实用户数据          抽象提炼           用户画像          │
│                                                             │
│   用户 A ──┐                            ┌──────────────┐   │
│   用户 B ──┤                            │  "小王"       │   │
│   用户 C ──┼──── 聚类 + 分析 ──────────>│  28岁/白领    │   │
│   用户 D ──┤                            │  追求效率     │   │
│   用户 E ──┘                            │  价格敏感     │   │
│                                         └──────────────┘   │
│   (海量零散数据)       (归纳共性)        (一个鲜活的人)     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 用户画像的核心价值

| 价值点            | 说明                                                   |
|:-----------------|:------------------------------------------------------|
| **对齐团队认知**  | 让产品、设计、开发、运营对"用户是谁"达成共识               |
| **指导产品决策**  | 做功能取舍时，回到画像思考"这个人需要吗？"                 |
| **提升同理心**    | 把抽象的"用户"变成有名字有故事的"人"，更容易共情           |
| **避免自我代入**  | 防止团队成员把自己的偏好当作用户需求                       |
| **沟通效率提升**  | 讨论时说"小王会怎么想"比说"用户会怎么想"更具体              |

## 二、如何创建用户画像

### 创建流程

```
  Step 1          Step 2          Step 3          Step 4          Step 5
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 收集数据 │───>│ 发现模式 │───>│ 聚类分组 │───>│ 构建画像 │───>│ 验证迭代 │
│         │    │         │    │         │    │         │    │         │
│ 访谈    │    │ 标记共性 │    │ 划分典型 │    │ 填充细节 │    │ 团队审核 │
│ 问卷    │    │ 标记差异 │    │ 用户类型 │    │ 赋予人格 │    │ 持续更新 │
│ 数据分析│    │ 寻找规律 │    │ (3~5类) │    │ 讲好故事 │    │ 实际检验 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

### Step 1：收集数据

数据来源包括但不限于：

- **定性数据**：用户访谈记录、客服反馈、社交媒体评论、用户社区帖子
- **定量数据**：用户注册信息、行为日志、问卷调查统计、App 数据埋点
- **二手数据**：行业报告、竞品用户评价、第三方数据平台

> 关键原则：画像必须基于真实数据，不能凭空想象。
> 即使时间紧张，至少也要做 5-8 次用户访谈作为基础。

### Step 2：发现模式

拿到数据后，逐条标记关键信息：

```
访谈记录标记示例：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
用户 A (女/26/设计师):
  [动机] "我主要在通勤路上逛" ← 碎片化场景
  [痛点] "搜索结果太多了选不过来" ← 决策困难
  [行为] 每周打开 App 5-6次 ← 高频用户

用户 B (男/31/程序员):
  [动机] "有明确需要才会上去买" ← 目的性强
  [痛点] "不想看推荐，直接搜" ← 效率优先
  [行为] 每月打开 App 2-3次 ← 低频用户
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Step 3：聚类分组

根据标记的信息，将用户按关键维度分组。常用维度：

- **使用动机**（为什么用？）
- **使用频率**（多久用一次？）
- **核心痛点**（最大的不满是什么？）
- **消费能力/意愿**（愿意花多少钱？）
- **技术熟练度**（对数字产品的使用能力）

> 提示：通常分成 3~5 个典型群组即可，太多反而失去聚焦。

### Step 4：构建画像

为每个群组创建一个"代表人物"，赋予以下信息：

- 基本信息（姓名、年龄、职业、所在城市）
- 个性特征与生活方式
- 使用场景与目标
- 痛点与需求
- 一句话引言（Quote）

### Step 5：验证迭代

- 让团队成员审阅画像，看是否与他们接触的用户一致
- 把画像拿去跟真实用户对照，验证是否有代表性
- 随着产品和市场变化，定期更新画像（建议每半年 review 一次）

## 五、常见误区

### 误区 1：画像 = 人口统计信息

```
  错误的画像                          正确的画像
┌──────────────┐                ┌──────────────────────────┐
│ 女性          │                │ 32 岁全职妈妈             │
│ 25-35 岁      │   VS          │ 追求性价比                │
│ 一线城市      │                │ 害怕买到假货              │
│ 月入 1-2 万   │                │ 碎片化时间购物             │
└──────────────┘                │ 信赖社区口碑              │
  (只有 Who)                    └──────────────────────────┘
                                 (包含 Who + Why + How)
```

人口统计信息只是画像的"骨架"，真正有价值的是**动机、痛点和行为模式**。

### 误区 2：画像是一成不变的

用户画像应该是**活的文档**。市场在变、产品在迭代、用户群体在演化。
建议每 6 个月 review 一次画像，特别是在以下时刻：
- 产品有大版本更新
- 目标市场发生变化
- 用户数据出现显著变化趋势

### 误区 3：画像越多越好

3~5 个画像通常就够了。如果你有 10 个以上的画像，
说明分类维度需要优化，或者你可能把子类型也独立成了画像。

### 误区 4：只靠拍脑袋创建画像

> "我觉得我们的用户大概是这样的......"

没有数据支撑的画像是危险的。至少需要：
- 5~10 次用户访谈
- 或 100+ 份有效问卷
- 或扎实的数据分析基础

### 误区 5：创建了画像但没人用

画像不是做完就放进抽屉的文档。好的做法：
- 把画像打印出来贴在团队工位旁
- 每次需求评审时对照画像讨论
- 新成员入职时作为必读材料
- 在 PRD 中注明"该功能主要服务于 [画像名]"

## 六、实践练习

### 练习 1：为你常用的 App 创建画像

选择一个你每天都在用的 App（如微信、抖音、美团等），思考并写出：
1. 这个 App 可能有哪 3 类典型用户？
2. 为每类用户创建一份简单的画像（可以用上面的模板）
3. 你属于哪个画像？你身边的人属于哪个？

### 练习 2：画像驱动决策模拟

假设你是某电商 App 的产品经理，现在有两个需求待排优先级：
- A：增加 AR 试穿功能
- B：优化优惠券叠加计算器

参考上面的三个画像，思考：
1. A 功能主要服务于哪个画像？B 呢？
2. 如果只能做一个，你会选哪个？为什么？
