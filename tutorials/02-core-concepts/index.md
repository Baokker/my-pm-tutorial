# 第二章：核心概念与术语

> 掌握术语，是进入任何专业领域的第一步。产品经理的日常工作横跨商业、产品、技术、设计四大领域，如果听不懂同事在说什么，协作就无从谈起。

---

## 为什么 PM 需要掌握这些术语？

作为产品经理，你每天都要和不同角色打交道：

| 协作对象 | 典型场景 | 你需要听懂的术语类型 |
|---------|---------|-------------------|
| CEO / 业务负责人 | 汇报产品价值、讨论商业模式 | 商业术语 |
| 产品团队 / 运营 | 需求评审、数据复盘、版本规划 | 产品术语 |
| 工程师 / 架构师 | 技术评审、方案讨论、排期沟通 | 技术术语 |
| 设计师 / UX 研究员 | 设计评审、交互走查、体验优化 | 设计术语 |

如果你在需求评审会上不知道什么是「灰度发布」，在商业汇报中搞混了「GMV」和「Revenue」，在和设计师沟通时不理解「Design Token」，这些都会严重影响你的专业可信度和工作效率。

**好消息是：** PM 不需要成为每个领域的专家，但需要达到「能听懂、能提问、能判断」的程度。

## 学习建议

### 1. 不要死记硬背

术语的意义在于理解和运用，而不是背诵定义。建议你：

- **先通读一遍**，对每个术语有个大致印象
- **在实际工作/案例分析中**，遇到不懂的再回来查
- **尝试用自己的话**向别人解释一个术语，能讲清楚才算真正理解

### 2. 关注术语之间的关联

很多术语不是孤立的，它们之间存在紧密的逻辑关系：

```
获客成本 (CAC)  ──→  用户生命周期价值 (LTV)  ──→  LTV/CAC 比率
      │                       │                        │
      ▼                       ▼                        ▼
  降低 CAC 的方法        提升 LTV 的方法         健康比率 > 3:1
  (优化渠道, PLG)       (提升留存, 提升 ARPU)    (商业模型可持续)
```

### 3. 建立自己的术语卡片

| 术语 | 我的理解 | 在哪个场景遇到的 | 还有什么不清楚的 |
|------|---------|----------------|----------------|
| GMV | 平台总交易额 | 电商月报 | 和 Revenue 的区别？ |
| ... | ... | ... | ... |

## 四大领域术语的详细预览

### 商业术语 -- 与钱有关的一切

商业术语帮助你回答一个核心问题：**这个产品怎么赚钱、赚了多少钱、值不值得继续投入？**

PM 在以下场景必须使用商业术语：

| 场景 | 涉及的术语 | 你需要做什么 |
|------|----------|------------|
| 月度数据汇报 | GMV, DAU/MAU, Conversion Rate | 用数据讲清楚产品的增长状况 |
| 投放效果评估 | ROI, ROAS, CPC/CPA, CAC | 判断营销花费是否值得 |
| 用户价值分析 | LTV, ARPU, NPS, Retention Rate | 评估用户质量和产品价值 |
| 融资/商业计划 | TAM/SAM/SOM, Unit Economics, Burn Rate | 向投资人证明商业可行性 |

重点掌握 **LTV/CAC** 比率 -- 它是衡量商业模型是否可持续的黄金指标。

### 产品术语 -- PM 的专业语言

产品术语是你作为 PM 最核心的工作语言，贯穿产品的整个生命周期：

| 产品阶段 | 核心术语 | 你需要做什么 |
|---------|---------|------------|
| 发现机会 | Persona, 用户旅程, North Star Metric | 理解用户、定义产品方向 |
| 规划方向 | OKR, KPI, Roadmap | 设定目标、制定计划 |
| 定义需求 | PRD, User Story, Acceptance Criteria, Backlog | 将想法转化为可执行的需求 |
| 设计方案 | Wireframe, Prototype, MVP | 从低保真到高保真验证方案 |
| 开发上线 | Sprint, 灰度发布, Feature Flag | 管理开发节奏和发布风险 |
| 验证效果 | A/B Test, 漏斗分析, Cohort | 用数据验证产品决策 |
| 规模增长 | PMF, PLG, SLG, K-Factor | 找到增长引擎并持续放大 |

重点掌握 **MVP** 的思维方式 -- 它不仅是一种产品方法，更是一种思维习惯。

### 技术术语 -- 和工程师说同一种语言

PM 不需要会写代码，但需要达到「能听懂、能提问、能判断」的程度：

| 理解层级 | 对应的术语 | 达标标准 |
|---------|----------|---------|
| 基础架构 | 前后端分离, API, 数据库 | 知道一个需求会影响前端还是后端还是都要改 |
| 数据能力 | 埋点, 数据上报 | 能独立设计埋点方案，知道数据从哪来 |
| 性能体验 | FCP/LCP/TTI, 崩溃率, 响应时间 | 能看懂性能报告，知道什么指标不达标 |
| 发布流程 | CI/CD, Git, 灰度/蓝绿部署 | 理解发版流程，能参与发布决策 |
| 系统能力 | 缓存, CDN, 微服务, 负载均衡 | 知道这些技术能解决什么问题，和产品的关系 |

重点掌握 **API** 和 **埋点** -- 前者是系统协作的基础，后者是数据驱动的前提。

### 设计术语 -- 和设计师高效协作

理解设计术语能让你从「感觉不对」进化到「准确表达哪里不对、为什么不对」：

| 协作阶段 | 核心术语 | 你需要做什么 |
|---------|---------|------------|
| 需求讨论 | 信息架构, Persona, 用户旅程 | 和设计师对齐用户理解和信息组织 |
| 方案评审 | UI vs UX, 交互设计, 视觉设计 | 从体验和视觉两个维度评估设计方案 |
| 规范检查 | Design System, Design Token, 组件化 | 确保设计方案符合系统规范 |
| 适配讨论 | 响应式设计, 自适应设计, 无障碍设计 | 参与多端适配和包容性设计决策 |
| 开发走查 | 设计走查, 标注/切图, Auto Layout | 检查实现效果是否与设计稿一致 |

重点掌握 **Design System** -- 它是保证产品体验一致性的基石。

## 常见问题

**Q: 这些术语需要全部背下来吗？**

A: 不需要。第一遍通读时，对每个术语建立印象即可。之后在工作和学习中遇到不确定的术语，回来查阅就好。随着使用频率增加，你会自然记住最常用的那些。

**Q: 不同公司对同一个术语的定义可能不同吗？**

A: 是的，非常常见。例如「活跃用户」的定义（打开 App 算还是完成某个操作算？）每个公司都不一样。入职后优先了解公司内部的指标口径定义，比背诵教科书定义更重要。

**Q: 学完术语就能做 PM 了吗？**

A: 术语只是基础。掌握术语意味着你有了和各方协作的语言基础，但成为优秀的 PM 还需要产品思维、需求分析能力、项目管理能力、沟通协调能力等。这些内容会在后续章节中详细展开。
